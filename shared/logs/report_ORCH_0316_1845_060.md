### 1. 脚本说明

**文件**: `GiT/scripts/visualize_pred_target_alignment.py`

**核心方法**:
1. Hook `model.task_specific_heads['occupancy_prediction_head']._get_targets_single_based_on_bev`
2. 执行 `model.forward(mode='loss')` 触发训练路径的 target 生成
3. 从 hook 中捕获真实的 `targets_tokens [400, 30]`, `tokens_weights`, `pos_inds`
4. 用 `TargetDecoder` 反编 target token → 可读字段
5. 对同一样本执行 `model.test_step()` 获取 prediction
6. 用 `PredDecoder` 解析 prediction
7. 逐 cell/slot 对齐，输出 6-panel 图 + JSON 报告

**Target 反编逻辑** (从 `_get_targets_single_based_on_bev` 输出):
- 每个 cell 有 30 token = 3 slots × 10 tokens
- Slot 结构: `[marker, cls, gx, gy, dx, dy, w, h, th_group, th_fine]`
- Token → 值的解码:
  - `marker`: tok 180=NEAR, 181=MID, 182=FAR, 183=END
  - `cls`: tok - cls_start(169) → class_id (0-9)
  - `gx`: tok - grid_start(0) → BEV grid X
  - `gy`: tok - grid_start - Wg(10) → BEV grid Y
  - `dx/dy`: tok - diff_start(20) → cell offset bin (0-20)
  - `w`: tok - wh_w_start(41) → width bin × 0.2m
  - `h`: tok - wh_h_start(142) → height bin × 0.2m
  - `th_group`: tok - theta_group_start(184) → angle coarse (0-35, ×10°)
  - `th_fine`: tok - theta_fine_start(220) → angle fine (0-9, ×1°)

**Target/Pred 同一样本保证**: 对 `dataset[idx]` 构建 `data_batch`，先传入 `mode='loss'` hook targets，再传入 `model.test_step()` 获取 predictions，使用完全相同的 data_batch。

**示例命令**:
```bash

```

**输出目录**: `/home/UNT/yz0370/projects/GiT/ssd_workspace/viz_orch060/`
- `sample_XXX_6panel.png` — 3×2 面板图 (Image Target / Image Pred / Mismatch | BEV Target / BEV Pred / BEV Overlay)
- `sample_XXX_cell_report.json` — 逐 cell/slot 的 target vs pred 对齐 JSON
- `summary.json` — 各样本统计摘要

### 3. 发现的异常（target 链路 + pred 行为）

#### 异常 1: Prediction 完全模板化 (CRITICAL)
- 所有 pred FG slot 输出几乎相同的坐标: `cx≈0.155, cy≈0.255`
- 所有 pred FG slot 预测相同的类: `construction_vehicle` (cls_id=4)
- cls_score 接近 1.0 (0.97-1.00)
- **结论**: ORCH_059 @500 iter checkpoint 的 prediction 已完全 frozen，输出固定空间模板

#### 异常 2: Pred marker 使用 ignore_id (229) 而非 END (183)
- 对所有 target FG 所在的 cell，pred slot 的 marker 是 `229` (ignore_id) 而非 `183` (END)
- 这意味着 decoder 在 target FG 区域甚至没有产出有效的 marker token，而是输出 padding token
- **可能原因**: marker_init_bias 导致 NEAR/MID/FAR 被过度抑制，decoder 无法产出有效 marker

#### 异常 3: around_weight=0.0 导致大量 target 权重为 0
- 当前 config `around_weight=0.0`，只有 center cell 有非零权重
- tokens_weights 范围 [0.000, 2.000] (sample 0) 或 [0.000, 1.000] (sample 5/10)
- 大量 FG cell 的 weight=0 意味着它们在 loss 中完全不参与——但仍然被标记为 FG target
- **影响**: pos_inds 统计包含这些 weight=0 的 cell，但梯度贡献为零，造成 effective positive 极少

#### 异常 4: target token 反编中无歧义
- Target token 编码是单射的，反编回原始值无信息损失
- `_get_targets_single_based_on_bev` 的输出与 `gt_projection_info` 中的 `bev_bbox_norm` 字段完全一致
- 未发现 target 生成链路本身的 bug
