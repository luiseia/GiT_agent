## Q1 判决: 终止 vs 继续到 @15000 的权衡

### 继续到 @15000 的论点 — 全部驳回

**论点 1**: "@12000 bg_FA=0.278 和 off_th=0.128 都是历史最低, 模型仍在学习"

**驳回**: 模型确实在学习 — 学习拟合错误的标签。bg_FA=0.278 看似改善, 但在 35.5% 物体被标为背景的标签下, 这个 "改善" 的含义是: 模型学会了把有物体的 cell 标为背景。这不是进步, 是模型在更好地拟合错误目标。

**论点 2**: "LR decay @15000 可能带来质变"

**驳回**: LR decay 改变的是优化速度, 不改变优化目标。优化目标 (标签) 本身是错的 → decay 只会让模型更稳定地收敛到错误的解。在正确标签上做 LR decay 才有意义。

**论点 3**: "offset 指标 (off_th=0.128) 不受 overlap 影响"

**部分成立, 但不构成继续的理由**: offset 精度衡量的是已分配 cell 的空间误差。center-based 分配下, cell 中心对齐物体中心 → offset 自然较小。overlap 分配会给更多 cell, 其中边缘 cell 的 offset 会更大。因此 offset 指标在不同标签模式下**不可直接对比**。@12000 的 off_th=0.128 不能作为 "不要停" 的理由。

**论点 4**: "再跑 3000 iter 只需 ~5h, 何不等等"

**驳回**: 机会成本。这 5h 可以用来训练 ORCH_028 的前 3000 iter (正确标签)。ORCH_024 的剩余训练对后续实验没有任何信息增量 — 我们已经知道 center-based 标签是错的。

### 终止的理由 — 全部成立

| 理由 | 量化证据 | 判断 |
|------|---------|------|
| 35.5% 物体零 cell → 标为背景 | 500 帧采样统计 | **致命** — 1/3 标签错误 |
| 70.1% 投影 < 56px → 分配不足 | grid cell 56×56 px | **致命** — 绝大多数物体受影响 |
| reg=0 频率 28.6% | 最近 28 iter 统计 | **严重** — 30% 训练步浪费 |
| car_P 天花板 | peak 0.090, 6 个 eval 未突破 | **符合预期** — 标签缺陷压低 precision |
| bg_FA 系统性上升 | 0.199 → 0.407 趋势 | **符合预期** — 真 occupied cells 被标为 bg |

**结论: 终止 ORCH_024。6 个 eval 点已经足够建立 center-based baseline。**

## Q2 判决: ORCH_028 从零开始还是 resume @12000?

### 必须从零开始

**理由 1: 标签分布根本改变**

Center-based → overlap 改变了每个 frame 的正/负样本分布:
- 平均每物体 cell 数: center ~1.2 → overlap ~3-4 (估计)
- 正样本率: ~1.8% → ~5-7% (估计)
- 背景错标率: 35.5% → ~0%

模型在 @12000 已经学到: "很多看起来有物体的 cell 实际上是背景"。这个 learned prior 在新标签下完全错误。

**理由 2: sqrt balance 权重会变**

```python
# git_occ_head.py:L820-836
_class_weights[c] = 1.0 / _math.sqrt(cnt / _min_count)
```

每类的 cell count 变化 → `_min_count` 变化 → 权重比例全部重算。resume 后优化器 momentum 基于旧权重, 与新权重不匹配。

**理由 3: 干净对比**

从零训练 ORCH_028, @2000/@4000/@6000 可以直接与 ORCH_024 同点对比, 量化 overlap fix 的真实增益。resume 的话, 混杂了旧知识, 无法归因。

**理由 4: 21h 成本可接受**

ORCH_024 从零到 @12000 用了约 21h。ORCH_028 同样 ~21h 到 @12000。但 ORCH_028 的每一步都在正确标签上训练, 信息价值远高于 ORCH_024 剩余的 28000 iter。

**结论: 从零开始, 不 resume。**

## Q3 判决: reg=0 频率 28.6% 是否足以构成终止理由?

### 单独来看: 不充分, 但加强了终止论据

**reg=0 的含义**: batch 中所有 400 个 cell 都是纯背景 → loss_reg = 0 → 该步只有 cls loss → 模型没有学到任何空间定位信息。

**28.6% (8/28) 的统计可靠性**:
- 样本量 28 iter 偏少, 置信区间宽
- 但即使真实率是 20%, 仍然意味着每 5 步就浪费 1 步
- 这是 BUG-51 的直接后果 (物体零 cell → 整帧可能无正样本)

**reg=0 是否独立构成终止理由?**

不是。如果标签正确, 只是某些帧确实没有物体 (如停车场空帧), 20-30% 纯背景 batch 是可接受的。问题是这些 "纯背景" batch 实际上有物体, 只是被标签遗漏了。

**结论: reg=0 是 BUG-51 的症状, 不是独立终止理由。但它量化了 BUG-51 的训练效率损失。**

### 正确的部分
1. **BUG-51 发现和修复** — 这是整个项目迄今最重要的发现。500 帧采样统计 (70.1%, 35.5%) 量化有力。修复方案 (overlap) 简洁正确。

### 需要修正的部分
1. **审计请求中 "反对理由" 给了太多犹豫空间**: Conductor 列了 4 条反对理由, 但没有一条经得起 BUG-51 的严重性检验。在 CRITICAL BUG 面前不应该列出 "继续的理由" 来混淆决策。
2. **reg=0 不应作为独立终止理由**: 它是 BUG-51 的症状, 不是独立信号。将它作为理由 1 会误导优先级判断。正确的框架是: BUG-51 (根因) → reg=0 (症状) + car_P 天花板 (症状) + bg_FA 膨胀 (症状)。
3. **ORCH_028 应考虑的配置变更**: Conductor 没有提到是否在 ORCH_028 中同时调整其他参数。建议 ORCH_028 **只改标签 (overlap)**, 其他参数全部保持与 ORCH_024 一致。这样才能干净归因 overlap fix 的效果。

1. **offset 指标基线会变**: overlap 给更多 cell, 边缘 cell offset 更大 → off_th/off_cx/off_cy 可能不如 ORCH_024。这不是退化, 是测量标准变化。Conductor 应提前设置新的 offset 预期。
2. **bg_weight 可能需要调整**: 正样本率从 ~1.8% 升至 ~5-7%, 正/负样本比例变化 → bg_weight=2.5 可能偏高。但建议先用 2.5 训练, 观察 bg_FA 后再调。
3. **之前的决策矩阵全部失效**: 所有阈值 (peak_car_P > 0.08, > 0.12, > 0.15 等) 都是在 center-based 标签下校准的。overlap 标签下 car_P 基线会大幅变化, 需要在 ORCH_028 @4000 后重新建立决策矩阵。
