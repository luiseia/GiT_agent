# 审计判决 — ORCH_035 @12000 Val 结果

## 结论: PROCEED

继续训练到 @16000。car_P=0.100 是所有实验的历史最高值 (ORCH_024 峰值 0.090)。car_R=0.620 超越 @8000。决策树所有指标通过。但 bus 已接替 cone 成为新 FP 工厂 (922K FP, 超过 car 的 849K)，BUG-17 类别轮换仍在活跃。BUG-17 修复必须在 @16000 eval 后立即部署，不可再推迟。

## 1. car_R/P 双指标回升: 真实改善，但 BUG-17 跷跷板的高点

### 完整轨迹

| Iter (有效) | car_R | car_P | car TP | car 总预测 | car FP | FP/TP |
|-------------|-------|-------|--------|-----------|--------|-------|
| @6k (2k) | 0.233 | 0.082 | 35,239 | 429,183 | 393,944 | 11.2 |

@12k 的 car FP/TP 比 = 9.1，是整个训练过程中最低值。模型从 @10k 的 "46% cells 预测为 car" (1.2M) 收缩到 @12k 的 "39% cells" (943K)，同时 TP 增加 48% (63.5K→93.8K)。

**car 学到了更精准的预测边界**。这不是随机振荡的高点——FP 绝对量下降 (-25%) 同时 TP 增加 (+48%) 表明分类头确实在改善。

### 但 BUG-17 跷跷板仍在活跃

| Iter | 类别组成 | FP 主要来源 |
|------|---------|-----------|
| @8k | car + ped + truck + bus (4类) | car 1.01M (54%) |
| @10k | car + bus + **cone** (3类) | car 1.14M (60%), cone 449K |
| @12k | car + bus + truck (3类) | **bus 922K (45%)**, car 849K |

每 2000 iter，FP 最大来源在 car / cone / bus 之间轮换。总 FP 从 1.88M → 1.90M → **2.06M**，呈上升趋势。car 改善的 FP (-287K) 被 bus 恶化 (+603K) 抵消有余。

### bus 已成为最大 FP 来源

| | bus_R | bus_P | bus TP | bus 总预测 | bus FP | FP/TP |

bus_R 从 0.089→0.149 (改善)，但 bus_P 从 0.006→0.004 (恶化)。bus 总预测从 415K 暴涨到 926K — **每个正确 bus 预测伴随 232 个 FP**。

bus_gt=26,711，不算小类 (FG 的 7.9%)。但 per-batch sqrt 权重在 bus 样本少的 batch 中仍会放大其 loss，驱动模型更激进地预测 bus。

### bus FP 超过 car FP

@12k: bus FP (922K) > car FP (849K)。这是第一次非 car 类成为最大 FP 来源。说明 BUG-17 的影响不限于小类——**任何类别在 per-batch 权重波动中都可能成为 FP 工厂**。

## 9. BUG-17 修复方案优先级

### 推荐: 方案 A — Weight Cap (max_w=3.0)

**修改位置**: `GiT/mmdet/models/dense_heads/git_occ_head.py:832`

```python
# 当前
_class_weights[c] = 1.0 / _math.sqrt(cnt / _min_count)
# 修改为
_class_weights[c] = min(1.0 / _math.sqrt(cnt / _min_count), 3.0)
```

**优势**:
- 一行改动，最低风险
- 直接防止极端权重 (当前小类可达 10×+)
- cap=3.0 意味着最小类最多 3× 权重，足以促进学习但不会产生 FP 洪水
- 向后兼容 (大类权重不受影响)

**验证方法**: mini 数据跑 2000 iter，确认无坍缩且类别 recall 分布更均匀

### 方案 B — Dataset-level sqrt (备选)

用全数据集类别统计替代 per-batch 统计。计算一次固定权重，消除 batch 间波动。中等工作量 (需在 config 中传入 class_counts)。

### 方案 C — 关闭 sqrt (核选项)

`balance_mode='equal'`。消除所有类别权重差异。后果: 小类可能永远不激活。仅在方案 A/B 验证失败时考虑。

### 不推荐 EMA

EMA 平滑 per-batch 权重在概念上合理，但引入额外状态 (每类的 EMA 计数器)，增加代码复杂度。当 weight cap 一行就能解决时，EMA 过度工程。

## 发现的问题

### BUG-61: 训练中偶发 reg_loss=0.0 iterations

- **严重性**: LOW
- **位置**: `git_occ_head.py` loss 计算逻辑
- **现象**: iter 8030 和 iter 12030 出现 `occupancy_prediction_loss_reg: 0.0000`，间隔 4000 iter。同时 cls_loss 异常低 (0.39, 0.47)，总 loss 远低于均值。
- **日志证据**:
  - `L6123: Iter(train) [12030/40000] loss: 0.4686 cls: 0.4686 reg: 0.0000 grad_norm: 17.31`
- **分析**: reg_loss=0 意味着 batch 中无 foreground GT cells。但 nuScenes 平均每帧 ~56 FG cells (339K/6019)，8-frame batch 应有 ~450 FG cells。出现 0 FG 统计上极不可能，暗示数据 pipeline 偶尔产生空 batch 或 `is_real_car_gt` mask 意外清空。
- **影响**: 极低 — 2 次 / 8000 iter (0.025%)。不影响训练趋势。
- **修复建议**: 在 loss 计算中加 debug log: 当 `is_real_car_gt.sum() == 0` 时打印 batch 信息。确认是数据问题还是 mask 逻辑问题。优先级低于 BUG-17。
